## 2025-04-08 - Interactive Icon Accessibility
**Learning:** When turning decorative icons into interactive elements (e.g., an `<X>` inside a trigger button to clear a filter), simply adding `onClick` and `role="button"` is not enough for full accessibility. Screen reader users can interpret the role, but keyboard-only users cannot focus the element without `tabIndex={0}`, nor can they interact with it without an `onKeyDown` handler listening for 'Enter' or 'Space'. Furthermore, preventing the event from propagating (to avoid triggering parent containers, like Popovers) requires `e.stopPropagation()` and `e.preventDefault()` in both the click and keyboard handlers.
**Action:** Whenever applying `role="button"` to a non-interactive element like a `<span>` or an SVG icon, always pair it with `tabIndex={0}` and an `onKeyDown` event handler to ensure keyboard operability, and verify event bubbling behavior within complex nested components.
## 2025-04-09 - Radix UI PopoverTrigger Child Accessibility
**Learning:** When using Radix UI components like `PopoverTrigger` with the `asChild` prop, the child element must be an interactive element (like `<button>`) rather than a simple `<span>` or SVG icon. While the `asChild` prop merges props correctly, passing a non-interactive element fails to provide standard semantic accessibility. The element will not be inherently focusable via keyboard, and it will lack proper ARIA attributes unless manually added, hindering screen reader users. Additionally, relying on `onClick` on a `<span>` does not provide native keyboard triggers (Enter/Space).
**Action:** When implementing icon-only triggers inside `PopoverTrigger asChild`, always wrap the icon in a semantic `<button>` tag, apply `aria-label` to describe the action and current state, and include clear focus-visible styles (`focus-visible:ring-2`, etc.) to support robust keyboard navigation.
